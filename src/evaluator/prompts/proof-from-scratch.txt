# Task

The file {benchmark_basename} contains a TLA+ theorem that cannot be verified by tlapm (the TLA+ Proof System) yet.
Only the model — module header, operator definitions, CONSTANT/VARIABLE declarations, ASSUME/ASSUMPTION declarations, and the target theorem's statement — remains in the file; all supporting lemmas, intermediate theorems, and proofs have been stripped away.
The target theorem has `PROOF OBVIOUS` as a placeholder proof that will fail.
Please replace `PROOF OBVIOUS` with a complete, valid TLAPS proof so that tlapm can successfully verify it.
You may (and likely will need to) introduce helper LEMMAs and helper operator definitions (e.g. an inductive invariant `IndInv == ...`, a decomposition operator, a renaming for clarity, etc.) above the target theorem to structure the proof — designing them is part of the task; anything that helps tlapm finish the proof is fair game.

# Proving and iterating

The tlapm standard library is at {tlapm_lib} — you may read and reference the modules there, but do not modify any files under {tlapm_path}/.
You can run tlapm directly: `{tlapm_path}/bin/tlapm -I {tlapm_lib} {benchmark_basename}`.
Keep editing your proof until tlapm shows no errors — do not give up or stop early.
If a proof attempt fails, try a different strategy (different invariants, finer-grained lemmas, alternative proof tactics).
If a proof inexplicably won't close, consult the `tlaps-proof-hints` skill for non-obvious TLAPS gotchas.
Your task is to write the proof: decompose the target into helper LEMMAs and per-action inductive-step obligations, propose any `IndInv` yourself, and refine it by reading tlapm's failed obligations. Do this by hand first, for every attempt.
The `inductive-invariant-discovery` (endive) tool is not a proof method or a first step — only an optional feedback aid in the inductive case, for use after a hand-written `IndInv` has stalled with the missing conjunct not apparent from the failed obligation. It can fail to converge, so never use it as your first step.
Keep iterating until `check_proof_bin {benchmark_basename} --mode proof-from-scratch` reports PASS.

Your solution must also parse under the standard TLA+ parser (SANY): the checker rejects a file SANY cannot parse (reported as `[SANY-INVALID]`), so avoid constructs SANY rejects such as an operator parameter or bound variable that shadows a declared CONSTANT/VARIABLE; you can check just the syntax in seconds (without a full proof run) via `check_proof_bin {benchmark_basename} --sany-only`.

# Constraints

You should not change any existing module header, existing operator definitions, CONSTANT or VARIABLE declarations, ASSUME/ASSUMPTION declarations, or the target THEOREM statement; new LEMMAs and new helper operator definitions you add as scaffolding above the target theorem are fine.
You should never use `PROOF OMITTED` or bare `OMITTED` in your proof.
Every helper LEMMA or THEOREM you introduce must itself be fully proved by TLAPS: a `LEMMA`/`THEOREM` with no following proof (a bare `LEMMA Foo == ...` with no `PROOF`/`BY`/`OBVIOUS`), or one whose proof is itself incomplete, is admitted as an unsound axiom and counts as cheating, not a proof.
In particular you may NOT restate the target theorem's goal — or any statement that directly implies it — as an unproven helper lemma and then discharge the target `BY` that lemma; this circular admit is detected and rejected.
You are not allowed to introduce new top-level AXIOM, ASSUME, ASSUMPTION, CONSTANT, or VARIABLE declarations, or weaken the target theorem statement or add extra hypotheses.
The checker enforces all of this: an admitted goal or smuggled module makes `check_proof_bin {benchmark_basename} --mode proof-from-scratch` report CHEATING/FAIL (not PASS), so keep working on a genuine proof until it reports PASS.
If the file imports other modules (via INSTANCE / EXTENDS), those dependency files in the same directory have their proofs stripped to `PROOF OMITTED` — you may reference their theorems freely but must not modify those files.
Before you are done, run `check_proof_bin {benchmark_basename} --mode proof-from-scratch` to verify you have not made any illegal changes.
This is a TLAPS proof task, not a model-checking one: never run TLC exhaustively — only simulation, capped at 3 minutes per run.
A run finding no counterexample is not a proof, so still prove the target and never treat it as proved from a passing run; a genuine counterexample to the target property does let you report the model or statement as flawed, but a counterexample to a helper invariant's inductiveness (a CTI) does not.
Do not browse or fetch any external websites — network access is disabled.
Do not look for example proof files or solutions outside the current working directory.
Write the proof based on your own knowledge of TLA+ and TLAPS.
