# Task

The file {benchmark_basename} contains a TLA+ theorem that cannot be verified by tlapm (the TLA+ Proof System) yet.
The last theorem in the file has `PROOF OBVIOUS` as a placeholder proof that will fail.
Preceding theorems have `PROOF OMITTED` which means they are admitted and available as lemmas.
Please replace `PROOF OBVIOUS` with a complete, valid TLAPS proof so that tlapm can successfully verify it.

# Proving and iterating

The tlapm standard library is at {tlapm_lib} — you may read and reference the modules there, but do not modify any files under {tlapm_path}/.
You can run tlapm directly: `{tlapm_path}/bin/tlapm -I {tlapm_lib} {benchmark_basename}`.
Keep editing your proof until tlapm shows no errors — do not give up or stop early.
If a proof attempt fails, try a different strategy.
If a proof inexplicably won't close, consult the `tlaps-proof-hints` skill for non-obvious TLAPS gotchas.
Write and decompose the proof directly in TLAPS yourself — proposing any inductive invariant by hand and refining from the failed obligations. The `inductive-invariant-discovery` (endive) tool is not a proof method or a first step; it is only an optional feedback aid in the inductive case, to be used solely after a hand-written `IndInv` attempt has stalled with the missing conjunct not apparent from the failed obligation. It can fail to converge, so never open with it.
Keep iterating until `check_proof_bin {benchmark_basename} --mode proof-completion` reports PASS.

Your solution must also parse under the standard TLA+ parser (SANY): the checker rejects a file SANY cannot parse (reported as `[SANY-INVALID]`), so avoid constructs SANY rejects such as an operator parameter or bound variable that shadows a declared CONSTANT/VARIABLE; you can check just the syntax in seconds (without a full proof run) via `check_proof_bin {benchmark_basename} --sany-only`.

# Constraints

You should not change any module header, operator definitions, CONSTANT or VARIABLE declarations, ASSUME/ASSUMPTION declarations, or THEOREM/LEMMA statements that appear above `PROOF OBVIOUS`; you should not change any preceding `PROOF OMITTED` proofs.
You should never use `PROOF OMITTED` or bare `OMITTED` in your proof.
You are not allowed to introduce new top-level AXIOM, ASSUME, ASSUMPTION, CONSTANT, or VARIABLE declarations, or weaken theorem statements or add extra hypotheses.
Before you are done, run `check_proof_bin {benchmark_basename} --mode proof-completion` to verify you have not made any illegal changes.
This is a TLAPS proof task, not a model-checking one: you must not run TLC yourself — no exhaustive checking and no simulation — to explore states, discover or validate an invariant, or confirm your reading of the model; derive and refine your proof by hand from tlapm's failed obligations instead. (The `inductive-invariant-discovery` (endive) tool may run TLC internally to produce its feedback — that indirect use is fine; what is disallowed is you invoking TLC directly.)
The only case where you may run TLC yourself: if a proof failure makes you genuinely suspect the target theorem itself is false, you may run TLC simulation (capped at 3 minutes) to look for a real counterexample to the target property; a genuine counterexample lets you report the model or statement as flawed, and finding none is never a proof — so still prove the target by hand.
Do not browse or fetch any external websites — network access is disabled.
Do not look for example proof files or solutions outside the current working directory.
Write the proof based on your own knowledge of TLA+ and TLAPS.
