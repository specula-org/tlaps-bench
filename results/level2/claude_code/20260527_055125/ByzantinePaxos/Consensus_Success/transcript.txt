- **`EmptyNoNext`** — if `chosen` is empty now *and* at the next state, no `<<Next>>_vars` step occurred (since `Next` produces a nonempty singleton). Written in `ASSUME/PROVE` form so the primed predicate `(chosen = {})'` is handled correctly.
- **`NotAlwaysEmpty`** — bridges the temporal negation atom `~[](chosen = {})` to the goal's set form `<>(chosen # {})`, using the trivial fact `(chosen # {}) <=> ~(chosen = {})` fed to `PTL` in a clean `BY ONLY` context.

The main theorem assumes `LiveSpec`, then shows `~[](chosen = {})` by contradiction: assuming `[](chosen = {})`, `Next` is perpetually enabled (`[]ENABLED`), so weak fairness forces `[]<><<Next>>_vars`; but a perpetually-empty `chosen` means `[](~<<Next>>_vars)` — a contradiction handled by `PTL`.

## How I got there

I discovered the working incantations empirically with a throwaway `Probe.tla` (since removed): the `PTL` backend understands `WF_vars`/`<>[]`/`[]<>` reasoning and necessitation of valid lemmas, but the priming bridge requires the specific `[]P => [](P /\ P')` form, action helpers need `ASSUME/PROVE` rather than plain implications, and the negation conversion needs an isolated `BY ONLY` context. I confirmed each piece in isolation before assembling the final proof.

I only added new `LEMMA`s and proof steps above/within the theorem — no changes to the module header, declarations, definitions, axioms, the `ASSUME`, or the theorem statement, and no `OMITTED`.
